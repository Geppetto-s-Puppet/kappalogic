# kappa_cpu — kappalogicのプリミティブだけで組んだ「数式上のCPU」

補助実験ノート。`kappalogic`本体には一切手を入れず、その上に外付けの
プロトタイプ(`kappa_cpu.py`)として実装した。付属の`kappa_cpu_demo.py`を
`python3 kappa_cpu_demo.py`で実行すればここに書いた検証結果は全部
再現できる。

## やったこと

`kappalogic.core`の`AND`/`OR`/`XOR`/`NOT`/`eq`だけを部品にして、以下を
if文を一切使わずに(すべてeq()による重み付き和のマルチプレクサとして)組んだ。

1. **全加算器 → n bit リップルキャリー加算器**(教科書のブール式をそのまま
   AND/XOR/ORに置換しただけ)
2. **6命令のALU**(ADD/SUB/AND/OR/XOR/NOT)。opcodeも`eq(opcode,k)`で
   デコードし、`Σ_k eq(opcode,k)*candidate_k`という重み付き和で選択する
3. **PC/ACC/CTRを持つ極小のフォン・ノイマン型トイCPU**。命令フェッチも
   `Σ_i eq(pc,i)*rom[i]`という同じマルチプレクサ形式で実装し、
   **PC自体を実数値の1個のレジスタ**として扱った。命令セットは
   `ADD, DEC_CTR, JNZ, HALT, LOADI_CTR, LOADI_ACC`の6個だけ
4. 実際に`CTR=5; ACC=0; loop: ACC+=CTR; CTR-=1; if CTR!=0 goto loop; halt`
   という**本物のJNZループ・プログラム**を実行し、`ACC=15`(=1+2+..+5)を
   bit-exactに得た(xi=1e-3、25サイクルのトレース全体)

これ自体は「CPUをtanhで書き直しただけ」で数学的には自明ですが、実際に
動くコードとして最後まで組んで検証したのは今回が初めてです。

## 見つけたこと(たぶん報告する価値があるもの)

### 1. 「回路の合成」は命題9/10のnaive-fold安全条件とは違う劣化の仕方をする

命題9(AND_n)・命題10(OR_n)は、**任意の実数値**を持つ入力をn項連鎖で
畳み込むとき、安全なxiの範囲がどう決まるかを扱っています。これを
「ビット幅nのリップルキャリー加算器」と「ループ長T(=CPUサイクル数)」
という2つの"連鎖の長さ"に当てはめて安全なxiの上限を実測したところ、
**どちらも、連鎖が伸びても閾値がほとんど縮まらない**という結果になりました。

```
ビット幅n:      2 bit -> safe xi <= 0.752
                4 bit -> safe xi <= 0.658
                8 bit -> safe xi <= 0.614
               16 bit -> safe xi <= 0.599
               32 bit -> safe xi <= 0.594   (n=2の約1.27倍縮んだだけで頭打ち)

ループ長(CPUサイクル数):
                ~12 cycles -> safe xi <= 0.289
                ~30 cycles -> safe xi <= 0.256
                ~96 cycles -> safe xi <= 0.256          (ほぼ完全に頭打ち)
```

これは命題9/10の直感(「連鎖が長いほど、途中に紛れ込む"境界ぎりぎりの値"の
確率が上がるので、安全マージンを稼ぐにはxiをさらに絞る必要がある」)とは
様子が違います。理由はたぶん単純で、**回路の各段に流れる値が、常に
「confidentな0か1」(あるいはPC/opcodeのような小さい整数インデックス)
そのものに強制的に揃えられている**からだと思います。命題9/10が問題にしていた
「任意の実数値・任意のスケールの入力が積み重なる」状況とは違って、
デジタル回路として使う限り、各ゲートの出力は次段に渡る前に必ず`reg`/`NOT`
で0/1近くに"再規格化"されるので、スケールが下流に向かって縮んだり
発散したりする余地がそもそもない、というのが実測から見えた仮説です。

**つまり**: kappalogicを「生の実数値を融合する」用途(OR_nのfold-vs-fuse)で
使うときは命題9/10の閉形式の安全条件が本質的に効いてくるが、「離散論理回路の
部品として、常にきれいな0/1(または整数インデックス)しか流さない」用途で
使う場合は、連鎖長に関係なくおおよそ一定の安全マージン(だいたいxi<0.25〜0.3
くらい、実測ではDEFAULT_XI=1e-3に対して200倍以上の余裕)で足りるようです。
これは命題9/10と矛盾するものではなく、**「fold-vs-fuse安全条件が本質的に
効いてくる領域」と「デジタル回路として使う限り気にしなくていい領域」の
境界線がどこにあるか**、という具体的な切り分けだと思います。厳密な証明では
なく実測(乱数探索+二分探索)なので、閾値の値そのものはまだラフです。

### 2. hard-modeの勾配消失が、CPU全体でもそのまま再現される(READMEの警告どおり)

PCを実数レジスタとして扱っているので、原理上「PCの初期値をわずかに
連続にずらしたら出力ACCがどう動くか」を数値微分できます。実際にやってみると、
安全に動作しているxiの領域(xi<=0.15くらいまで)では**勾配はほぼ完全に0**
でした。xi=0.2(安全境界ぎりぎり、命題9/10でいう"境界近傍"に相当)で
初めて非零の勾配(-0.0069くらい)が出ます。

これはREADME/dev_notes.mdが既に明記している「hard-modeはNP完全問題を
解く勾配降下の目的関数には向かない(勾配消失のため)」という限界の、
CPUというより大きな合成での再確認にすぎません。新しい発見ではなく、
**期待通りの結果が期待通りに起きた**という追試結果です。

同じ理由で、「単一のADD命令の入力を勾配降下で逆算する」小さいデモも
試しましたが(xi=0.3、A+11=20からAを求める)、局所解/プラトーに
捕まってきれいには解けませんでした(正解9に対し14で停滞)。これも
新しい発見ではなく、search.pyがhard/softモードを明示的に分離した
理由(dev_notes.md v0.2参照)を裏付けるだけの結果です。

## 誇張しないための注記

- 「CPU」と呼んでいますが、実態は**組み合わせ回路+ごく単純な状態遷移の
  シンボリック・シミュレータ**です。速度的には本物のCPUに一切勝てず、
  実用のプロセッサ設計とは無関係です。
- 上の「安全なxiが連鎖長に依らず頭打ちになる」という観察は、あくまで
  **2桁の乱数二分探索による実測**であり、命題9/10のような閉形式の証明では
  ありません。なぜ頭打ちになるのか(「毎段0/1に再規格化されるから」という
  説明)も直感的な仮説であって、厳密に示してはいません。
- 命令セットは6個の最小構成(ADD/DEC_CTR/JNZ/HALT/LOADI_CTR/LOADI_ACC)
  だけで、汎用レジスタファイルやメモリアクセス、複数命令の並列実行などは
  一切実装していません。「本物のCPUの理論的な雛形」というより、
  「eq()ベースのマルチプレクサでフォン・ノイマン的な制御構造が
  一応組めることを示した最小の存在証明」くらいの位置づけです。
- 勾配ベースの「プログラム探索」は、少なくとも素朴な数値勾配降下では
  今回は成功しませんでした。README自身の限界表明を追認しただけで、
  新しい可能性を開拓できたわけではありません。

## 先行研究はあるか(正直な調査結果)

「数式上のCPU」という発想そのものに先行研究があるかを調べた。結論:
**「微分可能なフォン・ノイマン機械」という枠組み自体は既によく研究されている
分野で、完全に新規というわけではない**。特に近いもの:

- **Neural Turing Machines** (Graves, Wayne & Danihelka, 2014) と
  **Differentiable Neural Computer** (Graves et al., *Nature* 2016,
  DeepMind) — 外部メモリへの注意機構による読み書きを微分可能にした、
  この分野の起点となる研究。
- **Neural Random-Access Machines** (Kurach, Andrychowicz & Sutskever,
  2015/ICLR2016) — **これが一番近い**。レジスタとメモリへのポインタを
  「全アドレスに対する確率分布」として表現し、ソフトアテンションで
  読み書きすることで、ポインタ参照ごとPC相当の構造を微分可能にしている。
  今回の実装で「PCを実数値のレジスタにし、`eq(pc,i)`の重み付き和で命令を
  フェッチする」という設計は、NRAMの「ソフトアテンションによるアドレス
  参照」と発想的にはほぼ同じことをしている。
- **Programming with a Differentiable Forth Interpreter** (Bošnjak et
  al., ICML 2017) — スタックベースの命令実行を丸ごと微分可能にした研究。
- **Neural Programmer-Interpreters** (Reed & de Freitas, 2016) — こちらは
  end-to-endではなく実行トレースの教師あり学習が必要な点で、NTM/NRAM系
  とは少し立ち位置が違う。

**kappalogicのCPUが上記と違う点**: NTM/NRAM系は「ソフトマックス的な注意
分布」でアドレス選択を実現しているのに対し、今回の実装は`kappalogic`
本体の`eq(x,k;ξ)=tanh((x-k)/ξ)²`をそのまま流用しているので、**ξ→0で
厳密な離散アドレッシングに収束することが、kappalogic本体の命題群
(命題1・命題21など)によって既に閉じた形で保証されている**、という点が
違う。NTM/NRAM系は基本的に「学習によってこの収束を獲得する」設計だが、
今回のものは「最初から解析的に収束が保証された対象を組んだだけ」で、
学習は一切していない(これは今回作ったものが"良い"という意味ではなく、
問題設定が違う、というだけ。NTM/NRAMは実際に例から入出力を学習できる
ことが売りで、今回のCPUは逆にそこが未検証)。

**誠実な結論**: 「微分可能なCPU/フォン・ノイマン機械」という大枠のアイデア
自体に新規性を主張することはできない(2014年のNTM以来10年以上の研究の
蓄積がある)。今回のプロトタイプの実質的な違いは、**注意機構を学習するの
ではなく、kappalogic本体の閉じた形の収束保証をそのまま流用してPC/opcode
選択を実装した**、という一点に絞られる。

## 続きにやったこと: 勾配ベースの「命令の逆算」を、ライブラリ公式の
   soft-mode + anneal_solveでちゃんと試す

前回の素朴な勾配降下(固定xi=0.3、hardモードのAND/XOR/ORをそのまま
微分)は局所解に捕まって失敗した。今回は`kappalogic.search`が公式に
用意している`anneal_solve`(熱方程式由来のスケジュール`xi=2√(Dt)`で
ξを大きい値から小さい値へ焼きなましながら勾配上昇する、3-SATデモで
実績のある手法)をそのまま使って再挑戦した。

**単一ゲートなら(AND/OR/XORいずれも)ほぼ100%成功する**(30/30、1変数の
探索)。しかし**リップルキャリー加算器を通して「A+11=20からAを逆算する」
問題は、ビット幅を増やすと急激に失敗するようになる**:

```
n=1 bit   正解が求まった割合: 19/20
n=2 bit                      : 15/20
n=3 bit                      : 10/20
n=4 bit                      :  2/20
n=5 bit                      :  0/20
n=6 bit                      :  0/20
```

単一ゲートの逆算(前回誤って「XORが本質的に難しい」と早合点したが、
1変数だけの探索ではAND/OR/XORどれも100%成功することを確認して訂正した)
はうまくいくのに、**ビット幅が伸びて全加算器が何段も連鎖すると、
ちょうどビット幅に比例するくらいのペースで成功率が線形に近い形で
崩れていく**。これは「安全なxiの上限はビット幅を増やしても頭打ちになる」
という前節の結果(順方向の正しさ)とは全く別の軸の話で、**順方向の
計算(forward)はビット幅に対して頑健なのに、逆方向の勾配探索
(inverse/program synthesis)はビット幅に対して脆い**、という非対称性が
今回はっきり見えた。

理由についての仮説(未検証): 加算器の各段は`XOR(XOR(a,b),cin)`のように
「1つ前の段の出力」を次段の入力に使うため、探索変数aの勾配情報が
桁上がりを通じて何段も伝言ゲームのように伝わる必要がある。単一ゲートの
実験で「1つのゲートなら勾配は素直に働く」ことは確認できたので、
崩れているのはゲート単体の性質ではなく、**探索対象の変数から出力までの
経路が伸びるほど、途中の飽和(tanh)を何度も通過することで実効的な勾配が
弱まっていく**という、通常のディープネットの勾配消失問題とほぼ同じ
現象だと考えられる。これは命題9/10のfold安全性とは別の劣化現象であり、
「正しく動く回路を組めること」と「その回路を勾配で"逆向きに解ける"こと」
は別の問題だ、というのが今回追加でわかったことだと思う。

## 誇張しないための追加の注記

- 上の「ビット幅に対して線形に近く劣化する」という観察も、n=1〜6の
  限られた範囲での実測(各20試行)であり、厳密なスケーリング則の
  導出ではない。
- 「単一ゲートは100%成功する」という結果も、今回のテストは
  target=1(1変数の探索)という単純な設定でのみ確認しており、
  一般のtarget値・2変数以上の同時探索では成り立たない可能性がある。
- 先行研究(NTM/NRAM/DNC)との比較は、実際にコードを動かして比較した
  わけではなく、論文の記述を読んでの位置づけの整理に留まる。
